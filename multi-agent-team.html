📐 系统架构 / System Architecture

用户

User / 用户

  • 提出任务需求
  • 接收最终报告
协调

Coordinator Agent

  • 与用户对话确认细节
  • 生成结构化任务描述
  • 发布到任务系统（带标签）
  • 跟踪进度并汇报结果
  • ❌ 不执行具体任务
调度

Dispatcher Agent

  • 监听 ready 状态任务
  • 根据类型标签匹配专家
  • 分配任务给匹配的专家
  • 处理专家拒绝的任务
  • ❌ 不执行具体任务
⬇️ 任务分配 Task Assignment ⬇️
专家

Expert Agents

  • dev — 编程专家
  • researcher — 研究专家
  • designer — 设计专家
  • writer — 写作专家
  • ✅ 可拒绝不匹配任务
审查

Reviewer Agent

  • 验收完成的任务
  • 拒绝时重新打标签
  • 不通过时反馈具体问题
  • 创建修改任务（最多3次）
系统

Kanban Task System

  • Hermes Kanban Board
  • todo → ready → running → done
  • 类型标签: type:code, type:research...
  • metadata 存储任务属性

🔄 工作流程 / Workflow

1

用户提出任务 → Coordinator 确认细节

用户描述需求，Coordinator 通过对话澄清目标、范围、约束、验收标准。模糊任务必须追问清楚。

2

Coordinator 发布任务到 Kanban（todo → ready）

任务包含：标题（[type] 前缀）、详细描述、类型标签（type:code/research/design/writing）、验收标准。

3

Dispatcher 匹配类型 → 分配给专家

Dispatcher 读取任务类型标签，从专家库匹配最适合的 Agent。type:code → dev, type:research → researcher。

4

专家评估 → 领取或拒绝

✅ 能力匹配 → claim 任务 → 执行 → 提交报告
❌ 不匹配 → 退回系统，说明理由 → 触发 Reviewer 重新打标签

5

Reviewer 审查验收

✅ 通过 → 标记完成 → 通知 Coordinator
❌ 不通过 → 写具体问题 → 创建修改任务 → 专家修改（最多3次循环）

6

Coordinator 汇总结果 → 汇报给用户

收集所有任务的执行结果、审查意见、产出物链接，生成最终报告。

⚖️ 负载均衡 / Load Balancing

根据专家当前工作负载分配任务，避免单点过载

✅ dev: running=0 (done=8)
⬜ researcher: running=0 (done=3)
⬜ writer: running=0 (done=0)
⬜ designer: running=0 (done=0)
→ Assign to: dev (lowest load)

🔥 优先级排序 / Priority

高优先级任务优先处理，同优先级按创建时间 FIFO

🔴 [high] API Rate Limiter (15:52)
🟡 [medium] Integration Tests (15:53)
🟢 [low] Rust Async Trends (15:53)
→ Process: high → medium → low

🔗 依赖管理 / Dependencies

检查任务依赖链，未满足时延迟调度，完成后自动提升

Task: Integration Tests
parent: API Rate Limiter
⏸️ BLOCKED: parent=ready
✅ PROMOTE: parent=done → ready

👥 专家容量规则 / Expert Capacity Rules

  • ✅ 同专家可并行处理多任务 (已验证)
    Same expert can handle multiple tasks in parallel (verified)
  • ⚠️ 软限制：3+ running 任务时触发负载均衡警告

    Single-expert types (e.g., code→dev) ignore capacity limit
